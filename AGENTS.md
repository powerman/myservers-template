# General rules for the project

## Project Context (Reference)

### Nature of the project

This project is for **operators**, not developers.
It configures and maintains personal servers —
installing and configuring software you didn't write (system services, self-hosted apps)
rather than deploying your own code.

Two use cases:

- **New server from scratch** — cloud-init (or manual) bootstrap script
  sets up the OS once (swap, SSH, basic firewall, Docker, automatic updates, alerting),
  then all further changes go through Docker containers.
- **Existing server** — adopt incrementally by adding services one by one
  into `docker-compose.yml`, without replacing the entire existing OS config.

### Multi-Server Awareness

This is an infrastructure-as-code repo managing multiple servers.
Always consider that a config may differ between primary and secondary servers —
ask before assuming something applies to both.

### Development Priorities

1. **Security** – Safely handle secrets, especially when using an AI agent.
2. **Simplicity** – Use only simple bash scripts and Docker Compose.
3. **Idempotence** — Container restarts must not break server configuration.
4. **Reliability** — Docker healthchecks everywhere, monitoring, alerts.

### Design Philosophy

Intentionally simple — no Ansible, Terraform, Kubernetes, or Docker Swarm.
Complexity of those tools is not justified for 2–5 servers managed by one person.

Deploy is always manual and interactive:
operator watches the `docker compose up --wait` output,
rolls back with `git checkout @~` + redeploy if healthchecks fail.
Redundancy (primary+secondary DNS/mail) covers the window of a failed deploy.

### Key system requirements

When adding or modifying system services, keep these constraints in mind:

- **nftables** must be Docker-compatible: pre-create Docker's filter chains,
  use priority to run before Docker's NAT.
  Container restarts must re-apply rules idempotently.
- **Postfix** shares `/var/spool/postfix` between host and container,
  so system daemons (cron, unattended-upgrades) can send mail
  through the containerized MTA.
  systemd failure handler also sends alerts via this shared Postfix.
- **WireGuard** entrypoints must trap EXIT to run `wg-quick down`
  for clean interface teardown on container stop.
- **All containers** must declare healthchecks
  (deploy uses `docker compose up --wait`).
- **Config templates** (`*.tmpl`) are rendered by dockerize at container start
  using environment variables, so configs are always regenerated from the current state.
- **Security updates**: OS packages update automatically (unattended-upgrades),
  Docker images are updated via Renovate PRs daily.
  Alpine packages inside Dockerfiles are updated via Renovate PRs daily.
- **Monitoring**: Netdata runs with host networking/PID access, monitors both OS and containers.
  Docker socket access goes through socket-proxy via unix socket.
- **Netdata priorities**: Primary use case is Telegram alerts;
  dashboard is secondary (inspected rarely, for investigations).
  Telemetry and crash reporting are disabled.

### Structure

- `.cache/` — Git-ignored directory for temporary files.
- `mise.toml`, `mise.lock`, `mise/` — Project tasks and tools managed by Mise.
- `fnox.toml` — Shared by servers fnox providers setup.

Servers:

- `srv/README.md` — Overview: IPs, hostnames, VPNs, port forwarding, URLs, quirks…
- `srv/primary/` — DigitalOcean; 1 vCPU, 1GB RAM; primary DNS and mail, VPN.
- `srv/secondary/` — OVHcloud US; 4 vCPU, 8GB RAM; secondary DNS and mail.
- `srv/homelab/` — Home server; Ollama.

Each server lives under `srv/SERVER/`.
Not all files are present in every server:

- `destination` — SSH target (e.g. `root@example.com`).
- `docker-compose.yml` — All services for the server.
- `mise.toml` — Mise config: fnox plugin/tools and env loading (secrets + host facts).
- `host-facts.sh` — Runtime host discovery (IPs, UIDs), provides fallback values in tests.
- `fnox.toml` — Encrypted secrets (managed manually by user using fnox CLI).
- `fnox.local.toml` — Auto-generated by mise task test:docker-build with fake secret defaults.
- `bootstrap.sh` — One-time OS setup script (see "Initial server setup" below).

### Tasks

Use these commands for corresponding tasks:

- `mise run lint` — runs all linters.
- `mise run fmt` — fixes formatting issues reported by linters.
- `mise run test` — runs all tests.
- `mise run test:docker-build SERVER` — runs `docker compose build` for the specified SERVER.
- `mise run generate:mermaid` — regenerates PNG diagrams from `srv/*.mmd` sources.

### Initial server setup (bootstrap script)

**Goal:** Minimal one-time OS setup to allow all further changes through docker containers only.

- Set up swap on servers with 1GB RAM or less.
- Set up hostname.
- Set up SSH: hardening, allow root login, AcceptEnv FNOX_AGE_KEY.
- Basic nftables setup compatible with docker and further container-based setup.
- Postfix on the host for accepting emails from daemons/cron
  (`/var/spool/postfix` shared with containerized Postfix later).
- Ensure any issues with OS updates or services are reported by email.
- Install and configure Docker and other tools used by deploy (mise).

NOTE: On non-cloud server, this script is uploaded and run manually on the first boot.

### Deploy

```sh
mise run deploy SERVER
```

This rsync's `srv/SERVER/` to `~/.myserver/` on the destination,
then copies the root `fnox.toml` as `~/.myserver/fnox.local.toml` on the server
(providing fnox provider config absent from the hierarchy on the server).
Finally, it sends `FNOX_AGE_KEY` via SSH `SendEnv` (requires `AcceptEnv` in sshd config)
and runs `mise exec -- docker compose build` followed by
`mise exec -- docker compose up --detach --remove-orphans --wait`.

Mise on the server loads `~/.myserver/mise.toml`,
which decrypts secrets via fnox and sources `host-facts.sh`.

### Secrets

**Design goals:** Commit encrypted secrets to git; minimize risk of leaking to AI agent.

Secrets are managed with [mise](https://mise.jdx.dev/) and [fnox](https://fnox.jdx.dev/).
Server secrets are encrypted and stored in `srv/SERVER/fnox.toml`,
using [age](https://github.com/FiloSottile/age) encryption.
Each `srv/SERVER/fnox.toml` contains secrets and behaviour (like `if_missing`).
Shared fnox config (`default_provider`, `recipients`) lives in the root `fnox.toml`
and is inherited via fnox's hierarchical merging.
On the server, deploy copies the root `fnox.toml` as `fnox.local.toml`
to provide the provider config that fnox needs for decryption.
Secrets are loaded (decrypted) into env vars
via the `_.fnox-env` directive in `srv/SERVER/mise.toml`.

**Decryption happens on the server**, not locally.
The age private key (`FNOX_AGE_KEY`) is sent via SSH `SendEnv`
during deploy and used by mise/fnox on the server.
No plaintext secrets are stored on disk — neither locally nor on the server.

**Isolation:** The root `mise.toml` sets `FNOX_AGE_KEY = false`,
which prevents leaking the age key to an AI agent through the environment.
However, fnox can still decrypt secrets using `~/.config/fnox/age.txt` directly.

Secrets required as files during `docker build` are passed via Docker BuildKit secret mounts.
They are defined in `docker-compose.yml` (`secrets:` + `build.secrets`)
and consumed in Dockerfiles with `RUN --mount=type=secret`.

### Testing

The `test:docker-build` task builds Docker images for each server
to validate Dockerfiles and compose configs.

**Fake defaults** for all secrets are injected into `srv/SERVER/fnox.local.toml` (gitignored)
by `test:docker-build` before each build.

The test script sets `FNOX_PROFILE=test` before `mise exec`.
`DEVEL=true` (set in root `mise.toml`) makes `host-facts.sh` use fallback values,
`FNOX_PROFILE` selects the fnox test profile with fake defaults.

### CI/CD

`.github/workflows/test.yml` runs `mise run ci` (a wrapper for fmt+lint+test tasks).
On PRs the test only builds changed servers;
on push to main plus daily it builds all.

Renovate updates Docker images, Alpine packages inside Dockerfiles, and GitHub Actions daily.

No deployment on CI; deployment is manual only.

### Setup Requirements

- [mise](https://mise.jdx.dev/) installed locally and on target servers.
- SSH access to target servers.
- `AcceptEnv FNOX_AGE_KEY` configured in sshd on target servers.
- Age private key in `~/.config/fnox/age.txt` on the operator's machine.

## Mandatory Rules

### Repository Safety

- DO NOT create, amend, squash, rebase,
  or otherwise modify existing commits.
- DO NOT switch branches.
- DO NOT perform any network git operations
  inside this repository
  (e.g. `git push`, `git pull`, `git fetch`).
- You MAY use `git stash` if necessary,
  but clean up after yourself.
- You MAY use `git restore` for reverting local changes.
- Do not delete, rewrite, or mass-modify files
  outside the explicit scope of the task.
- Avoid destructive shell commands
  (e.g. `rm -rf`, recursive operations)
  unless explicitly required.

### Coding Standards

#### Semantic Linefeeds (comments and documentation only)

Start each sentence on a new line.
Break long sentences at natural pauses —
after commas, semicolons, conjunctions,
or between logical clauses.
Do NOT hard-wrap to a fixed column width.
The goal is meaningful diffs:
one changed idea = one changed line.

NOTE: The above example does not mean you should break into very short lines as shown.

#### Documentation (markdown)

- Write new documentation in English.
- Avoid adding new documentation
  unless specifically requested by user.
- Update existing documentation together with code changes
  ONLY if otherwise existing documentation became incorrect.
- Keep lines within 96 characters.

#### Commenting

- Write new comments in English.
- Do not add redundant comments
  that restate obvious code behavior.
- Explain rationale, intent, trade-offs,
  and non-obvious behavior.
- Use full sentences in comments and documentation.
- Keep lines within 96 characters.

#### Formatting and Style

- General style is defined in `.editorconfig`.
- Shell script style is defined by `shfmt` flags in `mise.toml`.
- Always run `mise run fmt` to fix formatting.

#### Minimal Changes

Prefer minimal, targeted changes.
Do not add unnecessary wrappers, redundant config entries,
or settings already handled elsewhere.

### Keep Documentation in Sync

When any change affects information documented in `srv/README.md` or `srv/endpoints.mmd`,
update those files as part of the same change.

`srv/README.md` documents:

- Server addresses — IPs, interfaces, hostnames.
- DNS zones — primary/secondary NS mappings.
- VPN — endpoints, subnets, DNS servers, routes.
- Port forwarding rules.
- Services per server — service name, public ports, auth type.
- Caddy websites per server — URLs and auth type.
- Email — MX roles, submission/POP3S/IMAPS addresses, virtual domains, mail pipeline.
- Monitoring — Netdata URLs per server, Netdata Cloud.
- Backups — duplicity profiles, volumes, retention, what is not backed up.
- Operational notes — DO Reserved IP mechanics,
  .msmtprc in containers, nftables priority/chains, Courier userdb, manual extra setup.

`srv/endpoints.mmd` documents:

- Server subgraphs with interfaces, IPs and hostnames.
- Network connections between servers — protocols, ports, direction.
- External services — DO Spaces S3 (backup profiles), Netdata Cloud (connected instances).

After any change to `*.mmd` files,
run `mise run generate:mermaid` to regenerate the corresponding PNG diagrams.

### Bootstrap Scripts

Do NOT touch bootstrap scripts (`bootstrap.sh`) or initialization scripts
unless explicitly asked.
These are sensitive one-time setup scripts and should be left alone.

### Do Not

- Do not run `mise run deploy SERVER`.
- Do not read any files in `~/.config/fnox/`.
- Do not run `fnox` — in any form, from any directory.
- Do not run `mise` in `srv/SERVER` or its subdirectories (loads decrypted secrets into env).

Ask the user to perform any of the above actions manually if needed.

## Recommended Practices

Apply these unless the task explicitly requires otherwise.

### Key Patterns

**host-facts.sh**: Runtime host discovery script,
sourced automatically by `mise exec` via `_.source` in `srv/SERVER/mise.toml`.
Contains helper functions like `uid()`, `gid()`,
and `metadata()` (Cloud provider metadata API).
When `DEVEL` is set, uses fallback values for host-specific settings (IPs, UIDs).

**Dockerfiles**: Use `powerman/dockerize` for template rendering (`*.tmpl` → config files).
Alpine Linux base images.
Use exact versions for base image and main service or CLI tool,
use major versions for other packages (helper tools, 3rd-party dependencies) —
to get PR from Renovate only when important update is available.

**Networking**: Services use named Docker networks scoped per function (e.g. `mail`).
Network services (nftables, WireGuard) use `network_mode: host`.

### Debugging Guidelines

When debugging network or server issues,
focus on the user's stated root cause hypothesis first
before exploring alternative theories.
Do not chase tangential optimizations unless asked.

### Infrastructure-Specific Notes

When working with DNS, email (postfix/MX), or Docker networking,
double-check assumptions about record types (A vs CNAME),
routing paths (which server handles bounces),
and NAT return traffic before proposing changes.

### Tool / Shell Discipline

- Store temporary scripts in `.cache/`
  unless the language requires another location.
- Remove temporary scripts after use,
  unless the user is expected to run them manually.
- Avoid creating persistent artifacts
  unless required by the task.
- Prefer existing project tooling (`mise`)
  over ad-hoc commands.

## Common Gotchas

**Editing mise file tasks**: Scripts in `mise/tasks/` must be executable.
After editing any file in `mise/tasks/`, run `chmod +x` on it.

**Adding host-facts variables**: When adding a new variable to `srv/SERVER/host-facts.sh`,
also add it to the `environment:` section of the consuming service
in `srv/SERVER/docker-compose.yml` — otherwise the container won't see it.
